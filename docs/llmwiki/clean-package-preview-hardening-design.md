# Clean Package Preview Hardening Design

## Purpose

Y-08A is a docs-only hardening design for the existing clean-package preview
reports after the Y-07 text, Markdown, JSON, and regression checker work became
stable.

This design prepares later report-only improvements to package manifest
preview, package output diff prediction, and source coverage reporting. It does
not approve package generation, report file writing, or generated package
artifacts.

## Current Preview Baseline

- `Package manifest preview` exists in `scripts/clean_package_dry_run.py`.
- `Package output diff prediction` exists in `scripts/clean_package_dry_run.py`.
- The dry-run can output text, Markdown, and JSON.
- `scripts/check_clean_package_dry_run_reports.py` exists and checks the
  current report modes.
- No package files are generated by any current mode.
- `動画保存ツール_ローカル専用/` must remain absent.

## Non-Goals

Y-08A does not:

- change scripts;
- add tests;
- add CI;
- write report files;
- generate `manifest.json`;
- generate `NOTICE.txt`;
- generate `LICENSES/`;
- create `動画保存ツール_ローカル専用/`;
- copy license text;
- generate notice, license, inventory, HTML, or TXT guide output;
- build, package, or install anything;
- install or update dependencies;
- Docker pull/build is not performed;
- change backend, frontend, Docker, CI, package, or lockfile files;
- touch PR #1001 files, including `docker-compose.local.yml` and
  `docs/local-only.md`;
- handle cookie, token, secret, credential, private env, submitted URL, or
  private config values;
- add public hosting or ads;
- add 更新適用機能.

## Existing Preview Strengths

The current preview surface is already useful because it has:

- report-only behavior;
- an explicit no-generation note;
- source count and source list visibility;
- future output candidate visibility;
- package output diff prediction;
- human review required before generation;
- `legal_final: false`;
- secret, token, and cookie non-disclosure flags;
- text, Markdown, JSON, and report checker coverage.

## Preview Gaps

Gaps to address in later report-only work:

- Manifest preview may not yet expose enough per-output entry fields.
- Package output diff prediction may not group outputs by user-facing,
  developer-facing, or OS-specific sections.
- Source coverage may need clearer resolved and unresolved status labels.
- Notice source index mapping may need clearer component-to-output
  traceability.
- License review may need separate source, license, notice, and inventory
  statuses.
- Beginner guide output mapping may need explicit HTML and TXT output
  readiness.
- Developer review checklist may need package-time review fields.
- JSON report may need stronger object-level manifest entry arrays.
- Cleanup and rollback preview may need a clearer generated-root-only cleanup
  boundary.
- Cross-format preview data should stay consistent across text, Markdown, and
  JSON.

## Package Manifest Preview Hardening Candidates

A later report-only implementation may add richer manifest entry candidates.

Candidate manifest entry fields:

```text
package_relative_path
source_candidate
kind
target_os
target_arch
generated
required
include_reason
license_notice_required
review_status
safety_notes
source_status
output_status
human_review_required
```

Rules:

- Candidate fields are design only.
- `generated` must mean future planned output, not created output.
- Paths must be package-relative or repository-relative.
- Reports must not include private absolute paths.
- Missing source candidates should remain warnings in preview and blockers only
  before actual generation.
- Passing preview is not generation approval.

## Source Coverage Hardening Candidates

Future report-only source coverage may use these statuses:

```text
present
missing
candidate_only
source_draft
legal_not_final
package_time_review_required
not_applicable_this_phase
```

Use this vocabulary for:

- guide sources;
- notice sources;
- license sources;
- inventory sources;
- runtime selection sources;
- desktop shell candidates;
- package manifest candidates.

Rules:

- `present` only means the source candidate exists.
- `source_draft` means source material exists but is not generated output.
- `legal_not_final` means legal review remains unresolved.
- `package_time_review_required` remains true for exact artifacts, versions,
  license text, notices, and generated outputs.

## Cross-Format Requirements

Future hardening must keep preview data aligned across:

```text
text
markdown
json
```

Do not introduce fields in one format that contradict another. JSON may be more
structured, but text and Markdown must remain human-reviewable.

## Machine-Readable JSON Requirements

Future JSON hardening should:

- add a richer `manifest_entries` array;
- add a richer `output_diff_prediction` object;
- add source coverage statuses;
- keep required top-level fields stable;
- let consumers ignore unknown future fields;
- preserve valid JSON object stdout-only behavior.

## Sanitization And Privacy Rules

Reports must never print real:

- cookie values;
- token values;
- secret values;
- credential values;
- private env values;
- submitted media URLs;
- private config values.

Secret-like findings should only report:

```text
path
line number when safe
pattern family
sanitized message
```

## No-Generation Boundary

Y-08A is docs-only.

- No files are generated.
- No package output is created.
- No report file is written.
- No `manifest.json`, `NOTICE.txt`, `LICENSES/`, or guide output is created.
- Actual generation remains a later human-reviewed task.
